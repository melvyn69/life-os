Voici la v0.1 officielle.

010 — The Entity Model

Le modèle universel des entités vivantes

⸻

1. Préambule

Une vie humaine n’est pas composée de données.

Elle est composée de présences.

Une personne.
Un enfant.
Un lieu.
Un projet.
Une entreprise.
Une habitude.
Un souvenir.
Une passion.
Une peur.
Une valeur.
Un objet.
Un film.
Une ville.
Une recette.
Un chien.

Tous ces éléments sont très différents.

Mais dans la vie réelle, ils ont un point commun fondamental :

ils peuvent compter.

Ils peuvent revenir.
Ils peuvent évoluer.
Ils peuvent être reliés à d’autres choses.
Ils peuvent porter une histoire.
Ils peuvent déclencher une émotion.
Ils peuvent influencer une décision.
Ils peuvent devenir importants.

Life OS repose sur cette idée simple :

tout ce qui devient significatif dans la vie d’un utilisateur peut devenir une entité.

Une entité n’est pas une fiche.

Une entité n’est pas une ligne dans une base de données.

Une entité n’est pas un tag.

Une entité est une présence reconnue par le système.

Elle est la plus petite unité vivante de compréhension d’une vie humaine.

⸻

2. Définition d’une entité

Une entité est un élément identifiable de la vie de l’utilisateur, suffisamment significatif pour être reconnu, relié, enrichi et suivi dans le temps.

Elle peut être concrète :

* une personne ;
* un lieu ;
* un objet ;
* un animal ;
* une entreprise ;
* un document ;
* un événement.

Elle peut être abstraite :

* une passion ;
* une valeur ;
* une peur ;
* une habitude ;
* un objectif ;
* une idée ;
* une période de vie.

Elle peut être temporaire :

* un voyage ;
* une mission ;
* une décision ;
* une opportunité ;
* une conversation.

Elle peut être durable :

* un enfant ;
* une relation ;
* une maison ;
* une vocation ;
* une croyance profonde.

Une entité existe lorsque le système comprend qu’un élément mérite d’être traité comme quelque chose qui peut avoir une histoire.

⸻

3. Pourquoi les entités existent

Les logiciels classiques organisent les informations par format :

* notes ;
* fichiers ;
* messages ;
* calendriers ;
* tâches ;
* contacts ;
* photos.

Mais la vie ne fonctionne pas ainsi.

Une même personne peut apparaître dans des messages, des photos, des événements, des souvenirs, des projets, des décisions et des émotions.

Un même projet peut contenir des documents, des conversations, des lieux, des échéances, des personnes et des idées.

Une même passion peut influencer les habitudes, les voyages, les achats, les relations et les ambitions.

Life OS ne doit donc pas organiser la vie par format.

Il doit l’organiser par signification.

L’entité permet cela.

Elle transforme une masse d’informations dispersées en une structure compréhensible.

Elle permet au système de ne pas seulement stocker ce qui s’est passé, mais de comprendre ce que cela concerne.

⸻

4. La différence entre une donnée et une entité

Une donnée est un élément isolé.

Une entité est un élément qui porte du sens.

Une donnée dit :

“Rendez-vous jeudi à 15h.”

Une entité permet de comprendre :

* avec qui ;
* pour quel projet ;
* dans quel contexte ;
* avec quelle importance ;
* avec quel historique ;
* avec quelles conséquences possibles.

Une donnée est statique.

Une entité est évolutive.

Une donnée peut être oubliée sans conséquence.

Une entité peut devenir une partie durable de la mémoire personnelle.

La donnée décrit un instant.

L’entité relie les instants entre eux.

⸻

5. Ce qui compose une entité

Une entité vivante est composée de plusieurs dimensions.

Aucune dimension ne suffit seule.

Ensemble, elles permettent au système de comprendre progressivement ce que cette entité représente.

Une entité peut contenir :

* une identité ;
* un nom ;
* des alias ;
* un type ;
* une description évolutive ;
* des faits connus ;
* des souvenirs associés ;
* des événements liés ;
* des relations avec d’autres entités ;
* un niveau d’importance ;
* un niveau de confiance ;
* un historique ;
* des préférences ;
* des émotions associées ;
* des décisions passées ;
* des intentions futures ;
* des zones d’incertitude ;
* des règles de confidentialité ;
* des corrections utilisateur.

Mais une entité ne doit jamais être réduite à sa fiche.

Sa vérité n’est pas dans ses champs.

Sa vérité est dans son évolution.

⸻

6. Les grands types d’entités

Life OS doit pouvoir représenter toutes les dimensions importantes d’une vie.

Les types d’entités ne sont pas des silos.

Ils sont des portes d’entrée.

Une entité peut appartenir à plusieurs dimensions à la fois.

Les grands types initiaux sont :

* Personne ;
* Relation ;
* Projet ;
* Lieu ;
* Objet ;
* Animal ;
* Organisation ;
* Événement ;
* Souvenir ;
* Habitude ;
* Passion ;
* Valeur ;
* Objectif ;
* Santé ;
* Document ;
* Décision ;
* Idée ;
* Période de vie.

Ces types permettent de structurer la compréhension sans enfermer la réalité.

Un enfant peut être une personne, une relation centrale, une source de souvenirs, un repère de vie et une priorité future.

Une entreprise peut être une organisation, un projet, une source de revenus, un réseau de personnes, une responsabilité et une ambition.

Une ville peut être un lieu, un souvenir, un projet de vie, une émotion et une possibilité.

Une valeur peut être abstraite, mais influencer des choix très concrets.

Le modèle doit accepter cette richesse.

⸻

7. Le cycle de vie d’une entité

Une entité ne naît pas toujours complète.

Elle apparaît progressivement.

Son cycle de vie peut suivre plusieurs étapes.

1. Signal

Le système détecte qu’un élément revient ou semble important.

Il peut apparaître dans une phrase, une photo, un document, un calendrier, un message ou une décision.

À ce stade, ce n’est pas encore une entité confirmée.

C’est un signal.

2. Reconnaissance

Le système comprend que ce signal désigne quelque chose d’identifiable.

Il peut lui donner un nom provisoire.

Il peut le relier à un contexte.

Il peut commencer à le distinguer d’autres éléments proches.

3. Création

L’élément devient une entité.

Le système commence à conserver son histoire.

Il ne la traite plus comme une information isolée.

4. Enrichissement

L’entité gagne en profondeur.

Elle reçoit de nouveaux faits, liens, souvenirs, événements, préférences et corrections.

5. Stabilisation

L’entité devient fiable.

Le système comprend mieux son rôle dans la vie de l’utilisateur.

Elle peut être retrouvée, utilisée, reliée et rappelée au bon moment.

6. Évolution

L’entité change.

Une relation se transforme.
Un projet avance.
Une passion s’intensifie.
Une ville devient un futur lieu de vie.
Une habitude disparaît.

Le système doit préserver l’histoire sans figer l’identité.

7. Fusion, sommeil ou disparition

Avec le temps, certaines entités sont fusionnées, mises en sommeil ou supprimées.

Le modèle doit accepter que tout ne reste pas vivant pour toujours.

⸻

8. Les relations entre entités

Une entité seule a peu de valeur.

Sa puissance vient de ses relations.

Life OS doit comprendre que les éléments d’une vie sont reliés.

Une personne peut être liée à un projet.

Un projet peut être lié à une ville.

Une ville peut être liée à une ambition.

Une ambition peut être liée à une valeur.

Une valeur peut influencer une décision.

Une décision peut transformer une relation.

Les relations entre entités peuvent être :

* fortes ;
* faibles ;
* récentes ;
* anciennes ;
* explicites ;
* implicites ;
* confirmées ;
* incertaines ;
* positives ;
* négatives ;
* pratiques ;
* émotionnelles ;
* symboliques.

Le système ne doit jamais créer des liens de manière abusive.

Un lien n’est utile que s’il aide à mieux comprendre ou mieux accompagner l’utilisateur.

La qualité du Life Graph dépend moins du nombre de connexions que de leur justesse.

⸻

9. Les niveaux d’importance

Toutes les entités ne se valent pas.

Certaines sont centrales.

D’autres sont secondaires.

D’autres ne sont importantes que temporairement.

Le niveau d’importance d’une entité doit être dynamique.

Il dépend de plusieurs signaux :

* fréquence d’apparition ;
* intensité émotionnelle ;
* proximité temporelle ;
* impact sur les décisions ;
* lien avec les objectifs ;
* lien avec les relations clés ;
* mention explicite par l’utilisateur ;
* conséquences concrètes dans la vie réelle.

Une entité peut être très importante aujourd’hui et moins importante demain.

Une autre peut sembler faible au départ puis devenir centrale.

Le système doit donc éviter les hiérarchies rigides.

Il doit apprendre l’importance comme un mouvement.

⸻

10. Les niveaux de confiance

Une entité peut contenir des informations certaines, probables ou incertaines.

Life OS ne doit jamais confondre hypothèse et vérité.

Chaque entité doit pouvoir porter différents niveaux de confiance :

* confirmé par l’utilisateur ;
* observé directement ;
* déduit avec prudence ;
* probable mais non confirmé ;
* contradictoire ;
* obsolète ;
* à vérifier.

Cette nuance est essentielle.

Une intelligence personnelle devient dangereuse lorsqu’elle parle avec certitude de ce qu’elle ne sait pas vraiment.

Le système doit donc toujours pouvoir dire :

“Je sais.”

“Je pense.”

“Je ne suis pas sûr.”

“Cela a peut-être changé.”

“Tu m’as corrigé.”

La confiance n’est pas un détail technique.

C’est une condition de respect.

⸻

11. L’évolution dans le temps

Une entité vivante n’est jamais seulement ce qu’elle est aujourd’hui.

Elle est aussi ce qu’elle a été.

Une relation a une histoire.

Un projet a des phases.

Une habitude a une trajectoire.

Une ville peut passer de souvenir à désir, puis à projet, puis à réalité.

Une passion peut commencer comme curiosité et devenir vocation.

Le système doit donc conserver la temporalité.

Il doit comprendre :

* quand une information est apparue ;
* quand elle a été vraie ;
* quand elle a changé ;
* pourquoi elle a changé ;
* ce qui reste valable ;
* ce qui ne l’est plus.

Sans temporalité, la mémoire devient confuse.

Avec la temporalité, la mémoire devient intelligence.

⸻

12. La fusion des entités

Dans une vie réelle, le système peut rencontrer plusieurs fois la même entité sous des formes différentes.

Une personne peut être mentionnée par son prénom, son surnom, son email ou son rôle.

Une entreprise peut être nommée différemment selon les documents.

Un projet peut apparaître comme une idée, puis comme une tâche, puis comme un dossier.

Life OS doit pouvoir fusionner plusieurs représentations lorsqu’elles désignent la même réalité.

Mais la fusion doit être prudente.

Une mauvaise fusion peut déformer la mémoire.

Le système doit donc privilégier :

* les indices forts ;
* la confirmation utilisateur lorsque nécessaire ;
* la réversibilité ;
* la traçabilité ;
* la conservation des nuances.

Fusionner ne signifie pas écraser.

Fusionner signifie reconnaître que plusieurs fragments appartiennent à une même histoire.

⸻

13. La disparition d’une entité

Toutes les entités n’ont pas vocation à rester actives.

Certaines disparaissent naturellement.

Un ancien projet.
Une opportunité abandonnée.
Une habitude arrêtée.
Une relation devenue distante.
Un intérêt passager.

La disparition ne doit pas toujours signifier suppression.

Il existe plusieurs états possibles :

* actif ;
* discret ;
* en sommeil ;
* archivé ;
* obsolète ;
* supprimé ;
* oublié volontairement.

L’utilisateur doit toujours pouvoir demander l’oubli.

Mais le système peut aussi réduire naturellement la présence d’une entité lorsqu’elle n’a plus de rôle actif.

Une bonne mémoire personnelle ne garde pas tout au premier plan.

Elle sait laisser respirer.

⸻

14. Comment une entité devient vivante

Une entité devient vivante lorsqu’elle cesse d’être seulement reconnue et commence à être comprise.

Elle devient vivante lorsque le système sait :

* ce qu’elle est ;
* pourquoi elle compte ;
* à quoi elle est liée ;
* comment elle a évolué ;
* ce qu’elle représente émotionnellement ;
* quand elle doit être rappelée ;
* quand elle doit rester silencieuse ;
* comment elle influence la vie de l’utilisateur.

Une entité vivante n’a pas besoin d’être complexe.

Elle a besoin d’être juste.

Un enfant devient une entité vivante parce qu’il est lié à des souvenirs, des priorités, des lieux, des habitudes, des émotions, des décisions et un futur.

Un film peut devenir une entité vivante s’il est lié à une période, une personne, une émotion ou une inspiration.

Une recette peut devenir vivante si elle est liée à une tradition, une santé, une famille, un goût, un souvenir ou un projet.

Une valeur peut devenir vivante si elle influence réellement les choix de l’utilisateur.

La vie d’une entité ne vient pas de sa nature.

Elle vient de ses liens.

⸻

15. Le modèle universel

Le même modèle doit pouvoir représenter un chien, une entreprise, une ville, un enfant, un film, une recette ou une valeur.

Non pas parce qu’ils sont identiques.

Mais parce qu’ils partagent une structure fondamentale :

* ils peuvent être nommés ;
* ils peuvent être reconnus ;
* ils peuvent être reliés ;
* ils peuvent évoluer ;
* ils peuvent avoir une importance ;
* ils peuvent porter des souvenirs ;
* ils peuvent influencer des décisions ;
* ils peuvent être compris dans le temps.

Le modèle d’entité est donc universel.

Il ne réduit pas tout au même niveau.

Il donne à chaque chose une structure commune pour être comprise selon sa propre nature.

C’est cette abstraction qui permet à Life OS de rester simple en surface et profond en intelligence.

⸻

16. Ce que le modèle doit éviter

Le modèle d’entité ne doit jamais devenir un système froid de classification.

Il doit éviter plusieurs pièges :

* transformer la vie en base de données ;
* créer trop d’entités inutiles ;
* figer les personnes dans des profils ;
* confondre fréquence et importance ;
* confondre hypothèse et vérité ;
* relier abusivement des éléments ;
* rendre la mémoire anxiogène ;
* exposer trop de complexité à l’utilisateur ;
* donner l’impression d’une surveillance ;
* empêcher l’utilisateur de corriger ou d’oublier.

Une entité existe pour servir la compréhension.

Pas pour capturer la vie.

⸻

17. Le rôle du Companion

Le Companion utilise les entités pour mieux accompagner l’utilisateur.

Mais il ne doit pas parler comme un système de gestion.

Il ne dit pas :

“J’ai mis à jour l’entité Projet X.”

Il dit plutôt :

“Ce projet semble revenir souvent ces derniers jours. Tu veux qu’on le garde en tête ?”

Il ne dit pas :

“Relation forte détectée.”

Il dit :

“Cette personne semble importante dans ce sujet.”

Le Companion doit rendre l’intelligence naturelle.

Il doit cacher la mécanique sans cacher le contrôle.

L’utilisateur ne doit jamais avoir l’impression de gérer des entités.

Il doit avoir l’impression que le système comprend mieux sa vie.

⸻

18. Le principe de contrôle

Même si les entités sont détectées automatiquement, elles appartiennent toujours à l’utilisateur.

L’utilisateur doit pouvoir :

* voir ce que le système comprend ;
* corriger une entité ;
* renommer une entité ;
* fusionner deux entités ;
* séparer deux entités ;
* masquer une entité ;
* supprimer une entité ;
* demander pourquoi une entité existe ;
* demander pourquoi elle est reliée à une autre ;
* empêcher certaines mémorisations.

La compréhension personnelle n’a de valeur que si elle reste gouvernée par la personne concernée.

Life OS peut proposer.

Il ne possède jamais.

⸻

19. Conclusion

L’entité est l’unité fondamentale de Life OS.

Elle permet de transformer des informations dispersées en compréhension vivante.

Elle permet de relier les personnes, les projets, les souvenirs, les lieux, les passions, les habitudes, les valeurs et les événements.

Elle permet à la mémoire de ne pas être une archive.

Elle permet au Life Graph de ne pas être un schéma abstrait.

Elle permet au Companion de comprendre ce qui compte vraiment.

Une entité est ce moment où le système reconnaît qu’un élément de la vie de l’utilisateur mérite une histoire.

C’est la différence entre stocker une information et comprendre une vie.

Life OS ne construit pas une base de données personnelle.

Life OS construit un monde intérieur lisible, contrôlable et vivant.

Et chaque entité en est une présence.
