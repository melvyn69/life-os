## 3. Canonical Data Model

The canonical objects for Life OS v0.1 are:

- Capture
- Observation
- Entity
- Memory
- Briefing

Signals remain conceptual only in v0.1.

No signals table exists.

Signals may be used in product language, prompts, reasoning, or UI copy when helpful, but they must not become a persisted canonical object in v0.1.

## 4. Canonical Table Names

The only allowed tables are:

- captures
- observations
- entities
- memories
- briefings

No alternative names are allowed.

Do not create `capture_items`, `signals`, `entity_relations`, graph tables, join tables, audit tables, or automation tables in v0.1 unless a later canonical document explicitly changes this decision.

## 4.1 Canonical IDs

Every canonical object uses UUID.

IDs are generated by Supabase.

AI never generates IDs.

AI responses may reference existing IDs supplied by backend code, but AI must never create, invent, or persist new identifier values.

## 5. Canonical Processing Pipeline

Capture
↓

Observation
↓

Entity (status: suggested)
↓

Memory (status: suggested)
↓

User Validation
↓

Entity (status: active)
↓

Memory (status: active)
↓

Briefing

This pipeline is the core v0.1 loop. Any implementation that bypasses user validation for important durable memory is outside scope.

## 6. User Validation

Important data always requires user validation before becoming durable memory.

The more durable, sensitive, identity-related, relational, or consequential an interpretation is, the more explicit the validation must be.

AI output starts as a suggestion. The user decides whether it becomes active memory.

## 7. AI Responsibilities

AI may:

- extract
- summarize
- suggest
- classify
- link cautiously
- generate briefings

AI may never:

- invent facts
- decide alone
- overwrite memories
- perform external actions

AI must preserve uncertainty, avoid over-interpretation, and make its suggestions reviewable by the user.

## 8. Deterministic Responsibilities

The following responsibilities must remain deterministic:

- Statuses
- Permissions
- RLS
- Deletes
- Validation
- Dates
- Ownership

These are product trust boundaries. They must be implemented with predictable application logic, database rules, and explicit user actions, not generative behavior.

## 9. Confidence

Canonical confidence model:

- low
- medium
- high
- confirmed

No numeric confidence scale is canonical in v0.1.

`confirmed` means explicit user validation or an equivalent deterministic confirmation.

## 10. Sensitivity

Canonical sensitivity model:

- normal
- sensitive

No additional levels exist in v0.1.

Sensitive information includes data that may affect health, family, finances, religion, politics, sexuality, emotional vulnerability, conflict, identity, safety, reputation, or third parties.

## 11. Statuses

Canonical statuses:

- suggested
- active
- confirmed
- hidden
- archived
- deleted

No alternative persisted status names are canonical in v0.1.

UI labels such as inbox, analyzed, ignored, draft, or processing may describe screens or temporary interface states, but persisted canonical object statuses must use only the values listed above.

## 12. Deletion

Deletion is soft by default.

Objects are archived whenever possible.

Permanent deletion requires explicit user action.

The system must never silently delete important user data. When an object is removed from the active experience, its status should become `archived`, `hidden`, or `deleted` according to the user's action and the product context.

## 13. Authentication

Every user owns only their own data.

RLS is mandatory.

Every canonical table must include ownership and must enforce `user_id = auth.uid()` through Row Level Security policies.

No implementation path may expose one user's data to another user.

## 14. AI Architecture

OpenAI may only be called through Supabase Edge Functions.

Frontend never contains AI secrets.

The frontend may call Supabase and Supabase Edge Functions, but it must never call OpenAI directly.

## 14.1 Model Independence

The architecture must remain independent from a specific OpenAI model.

Database schema, frontend contracts, Edge Function interfaces, and validation logic must not depend on model-specific behavior, model-specific response quirks, or a hardcoded model as a product assumption.

Models may be changed behind the backend boundary without changing the frontend data model.

## 14.2 Prompt Ownership

System prompts are versioned.

Prompt templates belong to the backend.

Frontend contains no prompt logic.

The frontend sends user intent and canonical object identifiers to backend functions. Backend functions own prompt templates, prompt versions, model selection, response validation, and persistence.

## 15. JSON Contracts

Every AI response must be validated before persistence.

Malformed JSON must never corrupt data.

Edge Functions must parse, validate, normalize, and reject invalid AI responses before writing to Supabase.

If an AI response does not match the expected JSON contract, the system must fail safely and preserve the original Capture.

## 15.1 AI Failures

If AI fails:

the Capture remains.

Nothing is lost.

The user may retry.

AI failure must not delete captures, partially overwrite validated memory, or create active entities or memories.

## 16. Immutable History

Important objects should preserve history whenever possible.

Validated memories remain explainable over time.

The system should preserve the original Capture, the generated Observation, user validation, and relevant timestamps so a Memory can be traced back to why it exists.

Validated memories should not be overwritten in a way that destroys their prior meaning. Prefer correction, archiving, replacement, or history-preserving updates when possible.

## 18. Coding Principles

Simple over clever.

Readable over abstract.

Mobile-first.

No premature architecture.

Keep components small, dependencies minimal, and behavior easy to remove or extend.

## 19. Performance Principles

Optimize perceived speed.

Avoid unnecessary requests.

The user should be able to capture quickly, review clearly, and return later without feeling that the product adds operational weight.

## 20. Security Principles

Never expose OpenAI keys.

Never bypass RLS.

Never trust client data.

Validate all client input, validate all AI output, and enforce ownership at the database boundary.

## 21. UI Philosophy

Expose only what users need.

Hide internal AI complexity.

The interface should show captures, suggestions, validation choices, memories, entities, and briefings in human language. It should not expose database mechanics, prompt mechanics, or internal AI reasoning as product complexity.

## 21.2 Future Evolution

The codebase should remain extensible.

Do not introduce future complexity into v0.1.

Future features should be possible, but v0.1 must not contain scaffolding that makes the current product harder to understand, review, secure, or remove.

## 22. Backward Compatibility

Changes to the data model should preserve user data whenever possible.

Breaking changes require migrations.

Migrations must be explicit, reviewable, and designed to protect user-owned memory.

## 23. Final Rule

When in doubt:

prefer simplicity,

prefer caution,

prefer user control,

prefer documentation over assumptions.
