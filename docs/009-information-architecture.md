009 — Information Architecture

L’architecture d’un compagnon de vie personnel

⸻

1. Préambule

La plupart des logiciels organisent des informations.

Nous devons organiser une vie.

Cette différence change tout.

Une vie n’est pas composée de dossiers.

Elle n’est pas découpée en catégories indépendantes.

Elle n’est pas une succession d’écrans.

Une vie est un système vivant où chaque élément influence les autres.

Une rencontre peut modifier un projet.

Un voyage peut transformer une passion.

Une habitude peut améliorer une santé.

Un événement peut redéfinir une relation.

L’architecture du produit ne doit donc jamais imposer une structure artificielle.

Elle doit révéler naturellement la structure déjà présente dans la vie de l’utilisateur.

Notre objectif n’est pas de créer une application que l’on apprend à utiliser.

Notre objectif est de construire un environnement qui devient progressivement une extension naturelle de la mémoire, de la réflexion et de l’organisation personnelle.

⸻

2. Le principe fondateur

Une seule règle gouverne toute l’architecture.

L’utilisateur ne navigue jamais entre des fonctionnalités.

Il navigue entre des contextes de vie.

Les fonctionnalités existent.

Mais elles restent invisibles.

Ce que l’utilisateur perçoit, ce sont uniquement les éléments qui ont du sens pour lui.

⸻

3. Les grandes surfaces du produit

L’ensemble du produit repose sur quelques surfaces fondamentales.

Elles sont volontairement peu nombreuses.

Chaque surface possède une responsabilité unique.

Aucune ne doit empiéter sur les autres.

⸻

3.1 L’Entrée Universelle

C’est la porte d’entrée du système.

Tout commence ici.

L’utilisateur peut :

* écrire ;
* parler ;
* envoyer une photo ;
* partager un lien ;
* transmettre un document ;
* transférer un email ;
* poser une question ;
* raconter une idée.

L’utilisateur n’a jamais besoin de choisir où ranger une information.

Le système comprend.

Puis organise.

L’Entrée Universelle est le point de départ de toute interaction.

⸻

3.2 L’Inbox

L’Inbox représente ce qui attend l’utilisateur.

Elle n’est pas une boîte de réception classique.

Elle rassemble uniquement les éléments qui nécessitent une attention.

Par exemple :

* une décision ;
* une information à vérifier ;
* une tâche importante ;
* un document reçu ;
* une idée à clarifier ;
* une réponse attendue.

L’Inbox ne conserve pas la mémoire.

Elle représente uniquement le présent.

Lorsqu’un élément est traité, il quitte naturellement l’Inbox.

⸻

3.3 Le Briefing

Le Briefing répond à une seule question.

Que dois-je savoir maintenant ?

Il ne montre jamais tout.

Il montre uniquement l’essentiel.

Le Briefing rassemble :

* les événements importants ;
* les changements récents ;
* les rappels utiles ;
* les opportunités ;
* les alertes pertinentes ;
* les recommandations du Companion.

Le Briefing n’est pas un tableau de bord.

C’est une synthèse vivante.

⸻

3.4 Les Entités

Les entités constituent le cœur du produit.

Une personne.

Un projet.

Un lieu.

Une habitude.

Une passion.

Une entreprise.

Un objet.

Une valeur.

Un souvenir.

Chaque entité possède sa propre histoire.

Elle évolue dans le temps.

Elle entretient des relations avec d’autres entités.

L’utilisateur ne consulte pas des fichiers.

Il explore des éléments vivants.

⸻

3.5 La Timeline

La Timeline raconte l’histoire.

Elle représente la continuité d’une vie.

Elle relie naturellement :

* souvenirs ;
* événements ;
* projets ;
* décisions ;
* voyages ;
* apprentissages ;
* changements.

Elle permet de comprendre comment une situation est devenue ce qu’elle est aujourd’hui.

La Timeline ne sert pas à archiver.

Elle sert à donner du contexte.

⸻

3.6 La Recherche

La recherche est universelle.

Elle ne cherche pas uniquement des mots.

Elle comprend des intentions.

L’utilisateur peut demander :

Où ai-je rencontré cette personne ?

Quand ai-je commencé ce projet ?

Montre-moi toutes les idées liées au camping.

Que sais-tu sur mon fils ?

La recherche parcourt l’ensemble du Life Graph.

Elle ne dépend jamais d’une organisation manuelle.

⸻

3.7 Les Réglages

Les réglages ne concernent jamais la vie.

Ils concernent uniquement le fonctionnement du système.

Ils permettent notamment :

* gérer la confidentialité ;
* gérer la mémoire ;
* connecter des services ;
* exporter les données ;
* contrôler les notifications ;
* personnaliser certains comportements.

Les réglages restent volontairement séparés de l’expérience quotidienne.

⸻

4. Ce qui n’existe volontairement pas

Pour préserver la simplicité, certaines catégories disparaissent complètement.

Il n’existe pas :

* de dossiers ;
* de notebooks ;
* de projets fermés ;
* de bases de données visibles ;
* d’espaces de travail ;
* de tags obligatoires ;
* d’arborescences complexes.

Ces mécanismes existent éventuellement à l’intérieur du système.

Jamais dans l’expérience utilisateur.

⸻

5. Le cycle naturel d’une information

Toute information suit toujours le même parcours.

Étape 1

Elle entre.

Par l’Entrée Universelle.

⸻

Étape 2

Le Companion comprend.

Il identifie :

* les entités concernées ;
* le contexte ;
* l’intention ;
* le niveau de confiance.

⸻

Étape 3

L’information rejoint la mémoire.

Elle est reliée au Life Graph.

Elle enrichit les relations existantes.

Ou crée de nouvelles relations.

⸻

Étape 4

L’information évolue.

Elle peut être :

* confirmée ;
* corrigée ;
* enrichie ;
* oubliée ;
* devenir obsolète.

La mémoire reste vivante.

⸻

Étape 5

Le système la restitue.

Jamais parce qu’elle existe.

Mais parce qu’elle devient utile.

Le bon contexte.

Le bon moment.

La bonne personne.

⸻

6. Les flux naturels de l’utilisateur

Le produit ne doit jamais imposer un parcours.

Il accompagne les rythmes naturels d’une journée.

Les principaux flux sont :

Entrer une information.

Recevoir un Briefing.

Explorer une entité.

Rechercher un souvenir.

Prendre une décision.

Continuer un projet.

Retrouver un document.

Comprendre une situation.

Tous ces flux passent par les mêmes surfaces.

Jamais par des modules spécialisés.

⸻

7. Une architecture centrée sur les entités

Dans les logiciels traditionnels :

les informations appartiennent à des dossiers.

Ici :

les informations appartiennent à des relations.

Une photo peut appartenir simultanément :

* à un voyage ;
* à une personne ;
* à un souvenir ;
* à un lieu ;
* à une émotion ;
* à une période de vie.

L’utilisateur ne décide jamais de cette organisation.

Le système la construit progressivement.

⸻

8. Une architecture centrée sur le contexte

Le même élément peut apparaître à plusieurs endroits.

Non parce qu’il est dupliqué.

Mais parce qu’il possède plusieurs significations.

Un projet professionnel peut apparaître :

* dans le Briefing ;
* dans une personne impliquée ;
* dans une réunion ;
* dans une décision récente ;
* dans un objectif annuel.

Il s’agit toujours du même objet.

Vu sous différents angles.

⸻

9. Une architecture silencieuse

La meilleure architecture est invisible.

L’utilisateur ne doit jamais réfléchir :

“Où dois-je ranger cela ?”

Il pense simplement :

“Je veux te montrer quelque chose.”

Le système prend ensuite le relais.

Plus l’intelligence progresse,

plus l’interface disparaît.

⸻

10. Les principes de navigation

Toute navigation repose sur quelques règles simples.

Le contexte avant la hiérarchie.

Une information est retrouvée par son sens.

Jamais par son emplacement.

⸻

Une seule vérité.

Chaque élément existe une seule fois.

Le reste n’est que différentes vues.

⸻

Tout est relié.

Aucune information importante n’est isolée.

⸻

La profondeur est progressive.

L’utilisateur commence toujours par une vue simple.

Puis découvre davantage uniquement lorsqu’il en a besoin.

⸻

Aucun cul-de-sac.

Chaque élément ouvre naturellement vers d’autres éléments liés.

L’exploration reste fluide.

⸻

Toujours pouvoir revenir.

Le système conserve le fil de navigation.

Il respecte la manière naturelle dont l’utilisateur réfléchit.

⸻

11. Une architecture qui grandit avec la personne

L’architecture du produit ne change pas lorsque la vie devient plus complexe.

Qu’il s’agisse :

* d’un étudiant ;
* d’un parent ;
* d’un entrepreneur ;
* d’un retraité.

Les mêmes surfaces restent présentes.

Seule leur compréhension devient plus riche.

La simplicité de l’expérience est indépendante de la quantité d’informations mémorisées.

C’est la responsabilité du Companion de gérer cette complexité.

Jamais celle de l’utilisateur.

⸻

12. Ce que cette architecture rend possible

Grâce à cette architecture :

* une vie entière peut être explorée sans être fragmentée ;
* chaque information conserve son contexte ;
* chaque souvenir reste relié à son histoire ;
* chaque projet s’intègre naturellement aux autres dimensions de la vie ;
* chaque relation évolue avec le temps ;
* la mémoire devient organique plutôt qu’archivistique ;
* le produit reste simple malgré une intelligence toujours plus profonde.

L’utilisateur n’a plus le sentiment d’utiliser un logiciel.

Il a simplement l’impression que quelqu’un comprend enfin comment sa vie est réellement organisée.

⸻

Conclusion

L’architecture d’un compagnon de vie personnel ne consiste pas à organiser des écrans.

Elle consiste à rendre la complexité d’une existence humaine invisible.

L’utilisateur ne doit jamais apprendre où se trouvent les choses.

Il doit seulement vivre, penser, partager et avancer.

L’intelligence du système prend en charge tout le reste.

Lorsque cette architecture est réussie, l’interface cesse d’être un outil.

Elle devient le prolongement naturel de la mémoire, de la compréhension et de la continuité d’une vie.
