008 — MVP Definition

La première version capable de prouver la valeur d’un Companion de vie personnel

⸻

1. Préambule

Le plus grand risque d’un produit ambitieux n’est pas d’être trop petit.

C’est d’être trop complexe avant d’être utile.

La vision décrite dans les documents précédents est volontairement vaste.
Elle concerne toute une vie.

Mais une vie ne se construit pas en une version.

Le Companion non plus.

Le MVP n’a pas pour mission d’être complet.

Il a pour mission de démontrer une seule chose :

qu’une intelligence personnelle capable de comprendre progressivement une personne apporte une valeur suffisamment forte pour devenir une habitude quotidienne.

Tout le reste peut attendre.

Le MVP est donc un exercice de discipline.

Chaque fonctionnalité doit répondre à une seule question :

Est-elle indispensable pour prouver la valeur du Companion ?

Si la réponse est non, elle n’appartient pas au MVP.

⸻

2. L’objectif du MVP

Le MVP poursuit un objectif unique :

Devenir le premier endroit où une personne choisit naturellement de déposer les éléments importants de sa vie.

Pas parce qu’elle y est obligée.

Parce que c’est l’endroit où ces éléments prennent le plus de valeur.

Le produit doit réussir à transformer une simple capture d’information en compréhension progressive.

L’utilisateur ne doit pas avoir l’impression d’utiliser un outil.

Il doit avoir le sentiment que sa vie devient progressivement plus claire.

⸻

3. La promesse du MVP

À la fin de la première semaine d’utilisation, l’utilisateur doit pouvoir dire :

“Il commence déjà à me connaître.”

Après un mois :

“Il m’aide à retrouver des choses que j’aurais oubliées.”

Après plusieurs mois :

“Je ne pourrais plus revenir en arrière.”

Si cette progression n’apparaît pas, le MVP n’a pas atteint son objectif.

⸻

4. L’utilisateur cible initial

Le MVP ne cherche pas à satisfaire tout le monde.

Il vise un profil très précis.

Une personne qui :

* réfléchit beaucoup ;
* gère plusieurs projets personnels ou professionnels ;
* prend régulièrement des notes ;
* échange avec une IA ;
* souhaite mieux organiser sa vie sans utiliser dix applications différentes.

Cette personne est naturellement curieuse.

Elle accepte d’investir quelques minutes par jour si la valeur obtenue est nettement supérieure.

Elle est également capable de fournir des retours détaillés pour améliorer le produit.

Le MVP est conçu pour ces utilisateurs exigeants.

Pas encore pour le grand public.

⸻

5. Le scénario d’usage principal

Le MVP repose sur un scénario extrêmement simple.

Tout commence par une idée.

Ou une information.

Ou une photo.

Ou une conversation.

L’utilisateur ouvre le Companion.

Il dépose ce qu’il souhaite.

Puis il referme l’application.

Aucun classement.

Aucune catégorie.

Aucune organisation.

Le Companion se charge du reste.

Quelques heures ou quelques jours plus tard, cette information réapparaît lorsqu’elle devient utile.

C’est cette boucle qui doit créer la confiance.

Déposer.

Comprendre.

Relier.

Restituer.

⸻

6. Les surfaces produit minimales

Le MVP doit rester volontairement petit.

Il se compose de quelques surfaces seulement.

L’entrée universelle

Le point d’entrée principal.

Elle accepte naturellement :

* texte ;
* voix ;
* photo ;
* document ;
* lien.

L’utilisateur ne choisit jamais où ranger son information.

Il la dépose simplement.

⸻

L’inbox personnelle

Chaque nouvelle information apparaît d’abord dans une inbox.

Cette inbox représente les éléments récemment reçus.

Le Companion les analyse progressivement.

L’utilisateur peut :

* les consulter ;
* les corriger ;
* les compléter ;
* les supprimer.

L’inbox rassure.

Elle montre que rien n’est perdu.

⸻

La vue des entités

Le Companion transforme progressivement les informations en entités vivantes.

Par exemple :

* une personne ;
* un projet ;
* un lieu ;
* une passion ;
* une habitude ;
* un événement.

Chaque entité devient une page vivante.

Elle évolue avec le temps.

Elle raconte une histoire plutôt qu’une liste.

⸻

Le briefing quotidien

Chaque jour, le Companion propose un court briefing.

Jamais plus que quelques éléments réellement utiles.

Par exemple :

* ce qui mérite l’attention aujourd’hui ;
* ce qui évolue ;
* ce qui approche ;
* ce qui a été oublié ;
* ce qui pourrait être relié.

Le briefing n’est jamais une liste de notifications.

C’est une synthèse.

⸻

7. Le rôle de l’entrée universelle

L’entrée universelle est le cœur du MVP.

Elle remplace toutes les questions du type :

“Où dois-je enregistrer cela ?”

Le Companion doit être capable de recevoir des informations imparfaites.

Une phrase.

Une photo.

Un vocal.

Une idée.

Une capture d’écran.

Une URL.

Le système décide ensuite comment interpréter cette information.

Jamais l’utilisateur.

⸻

8. Le rôle de l’inbox

L’inbox constitue le sas de confiance.

Elle permet de voir ce qui vient d’entrer dans le système.

Elle rend visible le travail du Companion.

Elle évite la sensation de “boîte noire”.

Chaque élément peut être :

* confirmé ;
* corrigé ;
* enrichi ;
* ignoré ;
* supprimé.

L’utilisateur garde toujours le contrôle.

⸻

9. Le rôle des entités vivantes

Le MVP ne cherche pas encore à représenter toute une vie.

Il apprend simplement à reconnaître les éléments récurrents.

Chaque nouvelle information peut enrichir une entité existante.

Ou en créer une nouvelle.

Peu à peu, le Companion construit une compréhension de la vie de son utilisateur.

Sans effort supplémentaire.

⸻

10. La première version de la mémoire

La mémoire du MVP reste volontairement simple.

Elle mémorise uniquement ce qui apporte une valeur concrète.

Par exemple :

* les personnes importantes ;
* les projets ;
* les préférences explicites ;
* les événements ;
* les habitudes observées avec suffisamment de confiance ;
* les informations que l’utilisateur demande explicitement de retenir.

La mémoire reste transparente.

Chaque souvenir possède :

* une origine ;
* un niveau de confiance ;
* une date ;
* la possibilité d’être corrigé.

Le Companion préfère ignorer une information plutôt que mémoriser quelque chose de faux.

⸻

11. Les fonctionnalités indispensables

Le MVP doit permettre de :

* capturer une information depuis n’importe quel format courant ;
* analyser cette information ;
* identifier les principales entités ;
* enrichir progressivement la mémoire personnelle ;
* retrouver facilement une information ;
* consulter les pages des entités ;
* recevoir un briefing quotidien pertinent ;
* corriger la mémoire ;
* supprimer une information ;
* comprendre pourquoi une information est mémorisée.

Si une fonctionnalité ne contribue pas directement à cette boucle de valeur, elle peut attendre.

⸻

12. Les fonctionnalités volontairement exclues

Le MVP ne doit pas chercher à tout faire.

Sont exclus :

* la gestion complète des tâches ;
* le calendrier intelligent avancé ;
* les automatisations complexes ;
* les agents spécialisés ;
* les intégrations multiples ;
* les espaces collaboratifs ;
* les recommandations prédictives avancées ;
* la génération de plans complexes ;
* les workflows professionnels ;
* les tableaux de bord sophistiqués ;
* la personnalisation poussée de l’interface.

Ces fonctionnalités pourront apparaître plus tard.

Aucune n’est nécessaire pour prouver la valeur fondamentale du Companion.

⸻

13. Les critères de succès

Le succès du MVP ne se mesure pas au nombre de fonctionnalités.

Il se mesure au changement de comportement.

Les principaux indicateurs sont :

* l’utilisateur ouvre spontanément le Companion pour capturer une information ;
* il consulte régulièrement son briefing quotidien ;
* il retrouve facilement des informations importantes ;
* il corrige volontairement la mémoire lorsqu’elle se trompe ;
* il commence à faire confiance au système ;
* il ressent que le Companion comprend progressivement sa vie.

Le véritable indicateur est simple :

Le Companion devient un réflexe.

⸻

14. Les principaux risques

Le MVP peut échouer de plusieurs façons.

Le premier risque est la complexité.

Si l’utilisateur doit apprendre le produit, le produit a déjà perdu.

Le deuxième risque est la mauvaise compréhension.

Une mémoire erronée détruit la confiance beaucoup plus vite qu’elle ne se construit.

Le troisième risque est le silence.

Si le Companion ne restitue jamais de valeur, l’utilisateur cesse rapidement de l’alimenter.

Le quatrième risque est l’intrusion.

Une intervention mal choisie donne immédiatement une impression de surveillance.

Le cinquième risque est la dispersion.

Ajouter trop de fonctionnalités empêcherait de comprendre ce qui crée réellement la valeur.

⸻

15. Ce que nous devons apprendre

Le MVP n’a pas pour objectif de valider une technologie.

Il doit répondre à des questions fondamentales sur le comportement humain.

Nous devons comprendre :

* à quelle fréquence les utilisateurs déposent naturellement des informations ;
* quels formats sont les plus utilisés ;
* quels types d’entités émergent le plus souvent ;
* quels moments de restitution sont perçus comme les plus utiles ;
* combien d’interventions quotidiennes sont acceptables ;
* quels éléments les utilisateurs souhaitent réellement voir mémorisés ;
* quelles erreurs sont facilement pardonnées ;
* quelles erreurs détruisent immédiatement la confiance ;
* à partir de quel moment le Companion est perçu comme indispensable.

Ces apprentissages guideront toutes les versions suivantes.

⸻

16. Conclusion

Le MVP ne cherche pas à reproduire une application de notes, un gestionnaire de tâches, un calendrier ou un assistant conversationnel.

Il cherche à démontrer une idée beaucoup plus ambitieuse.

Qu’une intelligence personnelle capable d’écouter, de comprendre, de mémoriser avec discernement et de restituer au bon moment peut devenir un compagnon quotidien.

Si cette promesse est tenue avec une expérience simple, mobile et profondément fiable, alors le produit possède déjà son fondement.

Tout le reste — automatisations, intégrations, agents spécialisés, prédictions ou fonctionnalités avancées — ne sera plus une hypothèse.

Ce sera une évolution naturelle d’une relation de confiance déjà installée.
